## 🎯 AMENDMENT: Documentation (Phase 6) - NO BLOAT

**IMPORTANT CHANGE**: Phase 6 now has stricter documentation rules:

### The Rule: LEAN Documentation

✅ **UPDATE existing docs** (BUG_REGISTRY.md, FIX_LOG.md)  
❌ **DON'T create new docs** (unless absolutely essential)  
❌ **DON'T duplicate information** (single source of truth)  
✅ **ARCHIVE old docs** (move to archive/ folder when no longer relevant)  

### Before Creating ANY New Documentation, Check:

```
1. Does this info already exist elsewhere?
   → YES: Update that doc instead
   → NO: Continue to #2

2. Is this essential for operations?
   → NO: Don't document it
   → YES: Continue to #3

3. Can this fit in existing docs?
   → YES: Add to BUG_REGISTRY.md or FIX_LOG.md
   → NO: Continue to #4

4. Have we planned archival?
   → NO: Don't create new doc
   → YES: New doc approved (rare)
```

### What Gets Documented

✅ **DO document:**
- Bug fixes (BUG_REGISTRY.md)
- Work completed (FIX_LOG.md)
- Architecture changes (existing Architecture docs)

❌ **DON'T document:**
- Per-bug docs (use BUG_REGISTRY instead)
- Typo fixes (use code comments)
- Historical details nobody needs
- Duplicate information
- "Nice to have" docs

### Archival Strategy

**Old docs don't accumulate. Archive them:**

```
Active: docs/VISION_AND_GOALS.md
Archive: docs/archive/OLD_FEATURE.md (archived 2026-09-01)
Delete: After 6 months in archive
```

**Result**: Clean, lean, maintainable documentation.

**See**: DOCUMENTATION_AMENDMENT.md for full details.

## 🎓 The 4 Core Principles (Always Followed)

No matter what, these principles are applied to EVERY piece of work:

### ✅ Principle 1: Test-First Development
- Write tests BEFORE code
- All code tested with 80%+ coverage
- RED → GREEN → IMPROVE workflow

### ✅ Principle 2: One Commit Per Bug
- Each bug/feature = one clear, focused commit
- Commit message explains WHAT and WHY
- Easy to revert if needed

### ✅ Principle 3: Code Review Mandatory
- All code reviewed by fresh eyes
- CRITICAL/HIGH issues must be fixed
- MEDIUM issues fixed if quick
- Approval required before merge

### ✅ Principle 4: Security First
- No hardcoded secrets (API keys, passwords)
- No credential leaks in logs
- File permissions correct (0o600)
- Input validated
- No OWASP Top 10 vulnerabilities

## 🛡️ What Happens If Something Goes Wrong

**Scenario: Tests fail**
```
STOP. Do not merge.
→ Antigravity fixes tests or implementation
→ Re-runs tests until all pass
→ Re-requests code review
→ Only merges when ready
```

**Scenario: Security scan finds issue**
```
STOP. Do not merge.
→ Antigravity fixes security issue
→ Re-runs security scan
→ Only merges when clean
```

**Scenario: Preflight checks fail**
```
STOP. Do not deploy.
→ Antigravity fixes failing checks one by one
→ Re-runs preflight until 25/25 pass
→ Only deploys when ready
```

**Scenario: Production breaks**
```
IMMEDIATE ROLLBACK
→ Halt bot (stop new trades)
→ Revert to previous version (backup available)
→ Analyze what went wrong
→ Add test to prevent recurrence
→ Fix and test locally before redeploying
```

**You are always protected. Better to pause and fix than deploy broken code.**
